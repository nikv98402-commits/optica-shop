Target: local repo + production docs context for https://vilu.store
Mode: file/command audit. Browser screenshots unavailable because local gstack browse cannot start.

## Executive Summary

Overall DX: 6.1/10
TTHW: Competitive for an existing clone with dependencies already installed: typecheck ~3s, build ~3.25s, lint ~3s. Fresh-clone TTHW is inferred at 3-5 minutes because README gives only `npm install` + `npm run dev`, with no Node/npm version guidance or `.env.example`.

## Findings

1. Stale demo documentation

`DEMO.md` still says `VisionLux`, points at `https://nikv98402-commits.github.io/optica-shop/`, and describes old demo routes. This conflicts with `vilu.store`, ViLu positioning, GenEO pages, and latest release flow.

Impact: a new collaborator or partner following the demo script may open the wrong URL and present old messaging.

2. Obsolete frame asset path

`public/frames/README.md` recommends `imageUrl: '/optica-shop/frames/...'`, but production is now deployed at root domain. The correct example should be `/frames/...`.

Impact: copied examples will fail on `vilu.store`.

3. Missing `.env.example`

README documents `VITE_TALLY_FORM_URL`, `VITE_SUPABASE_URL`, and `VITE_SUPABASE_ANON_KEY`, but there is no `.env.example` file.

Impact: new developer has to infer variable names and safe defaults from README.

4. No single verify command

CI runs typecheck, lint, build, but locally the developer must remember three separate commands.

Recommended script: `"verify": "npm run typecheck && npm run lint && npm run build"`.

5. Release history exists in GitHub, but not in repo docs

There is no `CHANGELOG.md` or `RELEASE.md` explaining PR flow, deployment expectations, or rollback process.

Impact: harder to audit what changed between public releases without GitHub UI.

6. No contributor scaffolding

No `CONTRIBUTING.md`, issue templates, PR template, or support path.

Impact: fine for solo MVP, weak for partner/dev handoff.

7. Local working copy can drift from GitHub main

The local `.github/workflows/deploy-pages.yml` observed during this audit was stale versus GitHub `main`. GitHub main has the static route entries; local branch did not.

Impact: future work from stale branches can accidentally reintroduce fixed deploy behavior unless main is synced first.

## Recommendations

Priority 1:
- Update `DEMO.md` to ViLu/vilu.store and current CJM.
- Fix `public/frames/README.md` path examples from `/optica-shop/frames/...` to `/frames/...`.
- Add `.env.example` with safe empty/demo values.

Priority 2:
- Add `npm run verify` script.
- Add `CONTRIBUTING.md` with branch/PR/release workflow.
- Add `CHANGELOG.md` starting from the current public release.

Priority 3:
- Add `.github/PULL_REQUEST_TEMPLATE.md` with checks: typecheck, lint, build, canary, benchmark.
- Add issue templates for bug, content page, optics directory update.
- Add `docs/release-checklist.md` for DNS/GitHub Pages/Metрика/Search Console/rollback.

## Verdict

DONE_WITH_CONCERNS. The engineering loop is fast and workable, but docs are not yet handoff-grade. Fixing the stale docs and `.env.example` would lift DX from ~6/10 to ~7.5/10 quickly.
